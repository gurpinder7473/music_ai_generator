# AI Music Generator (LSTM + music21)

A complete pipeline that trains a neural network on MIDI music and generates
new music in the same style. Matches the 5-step workflow: collect data →
preprocess → build model → train → generate MIDI/audio.

## 1. Setup

```bash
pip install -r requirements.txt --break-system-packages
```

## 2. Collect MIDI data

Put `.mid` / `.midi` files in `data/midi/`. The more consistent the style
(e.g. all solo piano, all one composer/genre), the more coherent the output.
Some good free/public sources:

- **[MAESTRO dataset](https://magenta.tensorflow.org/datasets/maestro)** — ~200 hours of virtuosic solo piano performances, classical, freely licensed.
- **[Lakh MIDI Dataset](https://colinraffel.com/projects/lmd/)** — ~176,000 MIDI files across many genres.
- **[Classical Archives](https://www.classicalarchives.com/)** / **[Kunst der Fuge](https://www.kunstderfuge.com/)** — classical MIDI collections (check licensing per file).
- **[BitMidi](https://bitmidi.com/)** — a searchable library of free MIDI files across genres including jazz.

Start with 20–100 files of one style for a first experiment; scale up once
the pipeline works end to end.

## 3. Preprocess

```bash
python preprocess.py --midi_dir data/midi --out_dir data/processed --seq_len 100
```

This uses `music21` to parse each MIDI file into a stream of notes/chords,
turns each note into a token like `"C4"`, each chord into `"4.7.11"`, and
rests into `"REST"`. It then slices the whole token stream into overlapping
100-token windows (`X`) with the next token as the label (`y`) — the
standard "predict the next note" setup. Output is saved as `.npy`/`.json`/
`.pkl` files so you don't have to re-parse MIDI on every training run.

## 4. Train

```bash
python train.py --data_dir data/processed --epochs 50 --batch_size 64
```

Trains a stacked LSTM (`Embedding -> LSTM -> LSTM -> LSTM -> Dense(softmax)`,
defined in `model.py`) with dropout regularization. Checkpoints save
automatically to `checkpoints/`, and training stops early once loss
plateaus. On CPU, expect this to be slow (hours for a real dataset) — a GPU
is strongly recommended for anything beyond a tiny test run.

An optional GAN discriminator sketch is included in `model.py` for
reference, but LSTMs are the more stable and forgiving starting point for
symbolic music generation.

## 5. Generate music

```bash
python generate.py --model checkpoints/final_model.keras \
                    --data_dir data/processed \
                    --length 300 \
                    --temperature 1.0 \
                    --out output/generated.mid
```

This seeds the model with a random real sequence from the training data,
then repeatedly predicts the next token and feeds it back in, converting
the resulting token stream back into a `music21` stream and writing a
playable `.mid` file to `output/`.

- `--temperature` controls creativity: below 1.0 is safer/more repetitive,
  above 1.0 is more surprising (and more likely to sound chaotic).
- Open the resulting `.mid` in any DAW (GarageBand, Logic, Ableton, MuseScore)
  or a MIDI player to listen. To get a rendered audio file (`.wav`/`.mp3`),
  use a SoundFont synthesizer such as `fluidsynth`:
  ```bash
  fluidsynth -ni your_soundfont.sf2 output/generated.mid -F output/generated.wav -r 44100
  ```

## Project structure

```
music_ai_generator/
├── requirements.txt
├── preprocess.py     # MIDI -> token sequences (music21)
├── model.py           # LSTM (and optional GAN) architecture
├── train.py           # training loop with checkpointing
├── generate.py         # sampling + token-to-MIDI conversion
├── data/
│   ├── midi/           # <- put your raw .mid files here
│   └── processed/      # generated by preprocess.py
├── checkpoints/         # saved models during/after training
└── output/               # generated .mid files land here
```

## 6. Run as a Streamlit app

An interactive UI (`app.py`) wraps the whole pipeline into three tabs:
upload/preprocess → train → generate & download.

**Run locally:**

```bash
pip install -r requirements.txt --break-system-packages
streamlit run app.py
```

Then open the local URL Streamlit prints (usually `http://localhost:8501`).

**Deploy on Streamlit Community Cloud (free):**

1. Push this whole folder to a public (or private, on paid tiers) GitHub repo.
2. Go to [share.streamlit.io](https://share.streamlit.io), sign in, and click "New app".
3. Point it at your repo, branch, and `app.py` as the main file.
4. Streamlit Cloud will auto-install everything listed in `requirements.txt`.
5. Deploy — you'll get a public `*.streamlit.app` URL to share.

**Notes on hosting:**

- Free Streamlit Cloud sessions are CPU-only and have limited memory/time,
  so `app.py` defaults to small settings (short sequences, modest epoch
  counts) suitable for quick demos rather than large-scale training.
- For serious training, run `train.py` locally or on a GPU machine, then
  use the app mainly for the "Generate" step (you'd extend `app.py` to load
  a saved `.keras` model instead of training from scratch).
- The app lets users download the generated `.mid` file. Browsers can't
  play raw MIDI natively, so for in-browser playback you'd add a MIDI-to-
  audio conversion step (e.g. `fluidsynth` + a SoundFont) before returning
  audio bytes to `st.audio()`.

## Tuning tips

- **Sequence length** (`--seq_len` in preprocess): longer gives the model
  more context (more coherent phrasing) but is slower to train.
- **Model size**: 256-unit LSTM layers are a reasonable starting point;
  scale up if you have a large dataset and a GPU, scale down if overfitting
  (loss keeps dropping but generated output sounds repetitive/nonsensical).
- **Overfitting signal**: if loss goes very near 0, the model is memorizing
  rather than generalizing — add dropout, reduce model size, or add more
  training data.
- **Style separation**: if mixing genres (classical + jazz), consider
  training separate models per genre rather than one mixed model, since a
  single small LSTM will tend to blur styles together.
